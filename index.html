Product Intelligence — Deep Dive

Sharath · Analytical Team · AI Platforms Ops
Product Intelligence is the nerve center of the rollout. Sharath's team translates raw adoption data into actionable insight, documents where AI can be embedded as Agents & Assistants, and validates that what Development ships actually works. Without this lane, Development builds without direction and Rollout scales without signal.
🔍
1. Flow Analysis
Analyze the flows & document potential for improvements
Map current-state workflows for every department receiving AI tools. Identify where friction, latency, manual steps, or repeated tasks represent improvement opportunities.
Sharath's Team Delivers:
• Department workflow maps (current state vs. AI-augmented state)
• Friction point inventory by process and role
• Opportunity scoring matrix (impact × feasibility)
• Before/after time estimates for high-priority flows
📋
2. AI Implementation Docs
Document improvement needs via AI — Agents & Assistants
Translate flow analysis into structured requirements for Development. Each requirement specifies whether the solution should be an Agent (autonomous, multi-step) or an Assistant (user-facing, interactive).
Sharath's Team Delivers:
• Prioritized Agent & Assistant backlog (Jira/equivalent)
• Functional requirement specs (not technical — what, not how)
• User stories with acceptance criteria
• Business case / estimated ROI per build item
📏
3. Measurement Framework
Define leading & lagging indicators — measure usage, value & outcomes
Own the full measurement stack — from installation signals to business value. Define what "good" looks like at every stage and build the QuickSight dashboards that surface these signals automatically.
Leading Indicators (Predictive)
• Seat activation rate by department & wave
• Training completion rate (LMS module progress)
• Connector setup rate (Cowork)
• Time to first task / first dashboard
Lagging Indicators (Outcome)
• Hours saved per user per week
• Workflows automated (team-wide adoption)
• Business value / ROI (loaded cost × hours saved)
• Employee NPS on AI program
✅
4. QA of Agents & Assistants
Validate what Development ships before broad deployment
Product Intelligence owns quality assurance for every Agent and Assistant Development ships. This is a business-side QA — ensuring the tool solves the documented problem, not just that it runs without errors.
Sharath's Team Delivers:
• Acceptance criteria review (does it meet the requirement spec?)
• User acceptance testing with 3–5 pilot users before launch
• Output quality scoring rubric (accuracy, completeness, tone)
• Go/no-go sign-off before Rollout lane deploys broadly

Agent vs. Assistant — When to Build Which

For Sharath's Requirement Docs
🤖 Build an AI Agent when…
Autonomous · multi-step · runs without user prompting
✓ The task is repeatable and schedulable (daily/weekly trigger)
✓ It spans multiple systems (email + calendar + Jira + SharePoint)
✓ The steps are predictable and don't require mid-task human judgment
✓ Current process is fully manual and takes >30 min/week per person
✓ Output is a structured artifact (report, digest, alert, log entry)

Examples:
Weekly adoption metric digest auto-generated and emailed to managers
Daily ticket volume alert sent to Ops lead if threshold exceeded
Monthly impact report assembled from QuickSight + LMS + survey data
💬 Build an AI Assistant when…
Interactive · user-initiated · guides through a task
✓ The user needs to interact and provide context mid-task
✓ Output requires human judgment or approval at a decision point
✓ The task varies significantly in scope or inputs each time
✓ Use case is about augmenting a person, not replacing a process
✓ Output is a draft that a human reviews before sending/publishing

Examples:
Cowork skill that helps a manager draft a team adoption update email
Assistant that helps a non-technical user query their QuickSight data in plain English
Call prep assistant that pulls CRM + email context before a vendor meeting

Department Integration Plan — Claude Cowork & Amazon QuickSight

Cross-Enterprise Connectivity
Integrating Claude Cowork and Amazon QuickSight with each department's existing tools is what unlocks compound value — it turns both platforms from standalone apps into an interconnected AI fabric across the enterprise. This plan covers the key considerations, department-by-department tool mapping, and a phased 6-month action timeline.

Integration Considerations

Must Address Before Connecting
🔐 Security & Compliance
  • • SOC 2 Type II review for Anthropic (Claude) and AWS QuickSight before enterprise connector approval
  • • ITAR/export control assessment — satellite operations data may be restricted
  • • FedRAMP consideration for any government contract data
  • • Data Loss Prevention (DLP) policies: block PII, CPNI, and classified data from reaching AI models
  • • Audit logging required for all AI-tool interactions (who asked what, when)
  • • Penetration testing of connector endpoints before production go-live
🪪 Identity & Access
  • • SSO via Azure AD / Okta for both platforms — no standalone credentials
  • • OAuth 2.0 per-user connector authorization (Cowork: each user authorizes their own connections)
  • • Least-privilege access: QuickSight IAM roles scoped to department data only
  • • Row-level security (RLS) in QuickSight — Finance sees Finance data, not HR compensation data
  • • Service accounts for automated Agent tasks — separate from user accounts, auditable
  • • Access revocation workflow: auto-deprovision on employee departure
🗄️ Data Governance
  • • Data classification audit first: tag all enterprise data as Public / Internal / Confidential / Restricted before AI can access it
  • • Define which data tiers Claude can read, summarize, or act on
  • • QuickSight data catalog: every dataset must have an owner, refresh schedule, and sensitivity label
  • • Data residency: confirm AWS region (us-east-1 / us-west-2) aligns with your data policies
  • • Retention policy alignment: AI conversation logs + QuickSight query history
🏗️ Technical Architecture
  • • VPC endpoints for QuickSight → private data sources (no public internet traversal for sensitive data)
  • • API rate limit planning per connector (M365, Salesforce, ServiceNow all have throttling)
  • • SPICE capacity planning for QuickSight — pre-calculate dataset sizes across all departments
  • • Network proxy / firewall rules: Cowork desktop app needs outbound HTTPS to Claude API
  • • Data refresh cadence design: real-time vs. hourly vs. daily by dataset priority
  • • Fallback/degraded-mode behavior when connectors are unavailable
📋 IT Governance & Procurement
  • • IT Security review and approval required for each new connector type (Salesforce, ServiceNow, etc.)
  • • Third-party vendor risk assessment for each SaaS tool being connected
  • • Legal: Data Processing Agreements (DPAs) with Anthropic and AWS must be signed
  • • IT change management: connector additions go through change advisory board (CAB)
  • • Approved connector registry: IT publishes list of pre-approved connections; custom requests require additional review
  • • Budget approval for SPICE capacity expansion and any premium connector licensing
🔄 Change Management
  • • Department IT liaisons required for each integration wave — not just central IT
  • • Data owner sign-off before any departmental dataset is connected to QuickSight or Cowork
  • • User communication: what data Claude can see vs. what it cannot (important for trust)
  • • Connector onboarding guide per department — how to authorize, what's connected, how to revoke
  • • Escalation path for integration failures or unexpected AI behavior on connected data

QuickSight vs. Claude Cowork — Tool Selection Guide

For Power Users
The rule of thumb:  QuickSight answers "What happened and why?"  —  Cowork answers "What should I do about it, and can you handle it for me?"
📊 Amazon QuickSight
Enterprise BI & Analytics Platform
Structured, repeatable reporting on known data. Purpose-built for dashboards, data exploration, and sharing insights at scale across 8,000 employees.
Primary question it answers
"What happened? What does the data show?"
🤖 Claude Cowork
Intelligent Task Automation Platform
Unstructured, multi-step work across applications. Purpose-built for power users who need to execute on insights — not just visualize them.
Primary question it answers
"What do I need to do? Can you handle this for me?"

Power User Framework

3-Tier Certification Model
📊 QuickSight Power User Definition
Active
Active User: Logged in ≥2x/week, views existing dashboards, interacts with filters. Completes Foundations LMS module. Baseline engagement.
Advanced
Advanced User: Builds and publishes ≥2 original dashboards/month. Uses Q (natural language queries). Shares dashboards with ≥5 team members. Completes Intermediate LMS certification.
Power User
Power User / Champion: Authors enterprise-quality dashboards used by entire business units. Mentors peers. Demonstrates ≥4 hrs/month of documented time savings. Completes Advanced certification. Nominated to AI Champions network.
🤖 Claude Cowork Power User Definition
Active
Active User: Uses Cowork ≥3x/week. Has completed ≥1 multi-step task delegation. ≥1 connector authorized. Completed Cowork Foundations onboarding.
Advanced
Advanced User: Creates reusable skills or workflows. Uses ≥2 connectors per session. Delegates ≥5 tasks/week. Documents ≥1 use case in shared library. Completes Advanced Cowork LMS track.
Power User
Power User / Champion: Earns Cowork Power User Certification. Publishes ≥3 team-shared workflows. Self-reports ≥3 hrs/week savings. Participates in monthly town hall. Nominates peers. Acts as team AI ambassador.
